Fast trim vs Precise trim — which should I use?

Fast trim for almost everything. Lossless stream copy is instantaneous compared to re-encoding and produces a file with absolutely no quality degradation. The keyframe alignment shift (typically under 1 second) is invisible in practice for most use cases — removing a bumper from a recording, cutting dead air from the end of a video, splitting a long clip.
Precise trim when frame accuracy matters. If you're cutting to an exact frame — like keeping a clip that starts with a specific spoken word or cutting right before a particular moment — use Precise trim. It re-encodes the video, which takes longer and may slightly reduce quality, but cuts exactly where you tell it to.

How browser-based video trimming works

This tool uses FFmpeg compiled to WebAssembly (ffmpeg.wasm) — the same proven video processing engine used by professional tools like HandBrake and VLC, running entirely inside your browser tab. Your video file never leaves your device.

⚡
Stream copy mode
The default fast trim uses ffmpeg's -c copy flag — it repackages the existing encoded video and audio data without decoding or re-encoding anything. Essentially just cutting the container, not the content.
🔑
Keyframes explained
Video is compressed by storing full frames (keyframes) periodically and only the differences between frames in between. Stream copy can only start at a keyframe — so fast trim aligns to the nearest one.
🎯
Re-encode mode
Precise trim decodes the video fully and re-encodes from scratch. Frame-accurate, but slower and results in a small quality reduction (H.264 at high quality). Use only when exact cut points matter.
🔒
100% private
FFmpeg runs as a WebAssembly module in your browser tab. Your video data is processed in memory on your device and never transmitted to any server at any point.

Frequently asked questions

Fast trim uses stream copy mode which can only start at a keyframe — a complete video frame that doesn't depend on the frames before it. Most videos have keyframes every 1–5 seconds. When you set a start time that falls between keyframes, ffmpeg snaps to the nearest one. This is the trade-off for the speed and lossless quality of stream copy. If you need frame-accurate cuts, use Precise trim.
No. The tool uses FFmpeg compiled to WebAssembly running entirely in your browser. Your video file is read from your device into browser memory and processed there. Nothing is sent to any server. You can verify this by opening DevTools → Network tab and confirming no upload request is made when you click Trim.
WebAssembly in a browser tab is significantly slower than native video processing apps. On desktop computers, fast trim typically completes in 5–15 seconds. On mid-range phones, it may take 30–60 seconds. Precise trim (which re-encodes) can take several minutes on slower devices. This is an inherent limitation of browser-based video processing — the trade-off for running without any uploads. Keep the screen awake and the tab active while processing.
The tool supports MP4 (H.264/H.265), WebM (VP8/VP9), MOV, MKV, and AVI. MP4 with H.264 encoding is the most reliable format for both fast and precise trim. Some less common codecs inside MKV or AVI containers may not work — if trimming fails, convert the video to MP4 first using a desktop app, then trim here.
The tool accepts files up to 2GB. Processing large files (above 500MB) on lower-end devices or phones may be slow or cause the browser tab to run out of memory. For very large files, trim the rough section first at a larger size, then refine if needed.
