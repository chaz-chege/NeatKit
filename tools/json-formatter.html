Tips for working with JSON

Tree view is the fastest way to explore a deeply nested JSON object. Click ▶ to collapse nodes and focus on the parts you care about. Use it when the formatted text output is too long to navigate easily.
Minify before sending. Pretty-printed JSON can be 30–50% larger than minified due to whitespace. Minify before including JSON in API requests, config files, or anywhere size matters. Format it again here when you need to read it back.
The error message tells you exactly where the problem is. JSON parse errors include a position indicator — count characters from the start, or look for the common mistakes: trailing commas (not allowed in JSON), single quotes instead of double quotes, or unquoted keys.

How the JSON formatter works

This tool parses your JSON using JavaScript's built-in JSON.parse() — the same parser your browser uses for API responses. If the JSON is valid, it re-serialises it using JSON.stringify() with your chosen indent level, producing consistently formatted output.

The tree view builds a collapsible HTML representation of the parsed object, colour-coding each value type: keys in purple, strings in green, numbers in orange, booleans in blue, and null in red. This makes it easier to spot structure and data types at a glance without reading raw text.

Common JSON syntax errors and how to fix them

Most JSON errors come from a small set of mistakes. Here are the most common ones and what to look for.

ErrorWhat it meansFix
Unexpected token ,Trailing comma after the last item in an array or objectRemove the final comma before ] or }
Unexpected token 'Single quotes used instead of double quotesReplace all ' with " for both keys and string values
Unexpected token <key>Object key is not quotedWrap all keys in double quotes: key → "key"
Unexpected end of JSONMissing closing bracket or braceCheck that every { has a matching } and every [ has a matching ]
Unexpected token uundefined value in the JSONJSON doesn't support undefined — replace with null
Unexpected token /Comment in the JSONJSON doesn't support comments — remove all // and /* */ lines

JSON vs related formats

📋
JSON vs YAML
YAML is more human-readable (no quotes, indentation-based), supports comments, and is common in config files (Docker, Kubernetes, GitHub Actions). JSON is better for APIs and data exchange — every language has a JSON parser built in.
📄
JSON vs XML
XML uses opening and closing tags and is significantly more verbose. JSON is lighter and easier to parse. Most modern APIs have moved from XML to JSON. XML is still common in enterprise systems, SOAP APIs, and document formats like .docx.
⚡
JSON vs JSON5
JSON5 is a superset of JSON that allows trailing commas, single quotes, comments, and unquoted keys — making it friendlier for humans to write by hand. Not all parsers support it; standard JSON is safer for data exchange.
🔐
JSON in JWTs
JSON Web Tokens contain Base64-encoded JSON in their header and payload sections. Paste the middle section of any JWT into the Base64 decoder on Neatkit to read the raw JSON claims inside.
