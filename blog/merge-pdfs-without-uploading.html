How to merge PDFs without a website seeing your documents

Neatkit Team July 2026 6 min read

PDF merging sounds like a trivial operation. Combine two files into one. How complicated can it be? Complicated enough that a dozen well-funded companies have built profitable businesses around it — and the reason those businesses exist is that your files have to go through their servers to make it work.

Except they don't. They haven't had to for a few years now. The upload is a legacy of how these tools were originally built, and it persists because it's profitable, not because it's necessary.

This is the full picture: why the upload happens, what it means for your documents, and how to merge PDFs without sending a single byte to any server.

What most PDF mergers do with your files

When you upload a PDF to Smallpdf, iLovePDF, Adobe Acrobat online, or any of the major PDF services, here is the sequence of events:

Your file travels over HTTPS to their servers. It's stored temporarily on their infrastructure — typically in cloud storage like AWS S3 or Google Cloud Storage. Their backend software (usually a library like Ghostscript, PDFium, or a commercial equivalent) processes the file. The result is stored in the same cloud storage. You download it. The original and result are scheduled for deletion, usually within 1–24 hours depending on the service.

The HTTPS part is real security — your file is encrypted in transit. But once it lands on their servers, you're trusting their infrastructure, their employees, their security posture, and their deletion policies. Most of these companies are genuinely trustworthy. The point is that trust is required at all, and trust is not the same as certainty.

For a merge of two arbitrary PDFs — say, a recipe and a travel itinerary — none of this matters. For a merge of a signed NDA and a contract schedule, or a bank statement and a supporting letter for a visa application, it matters quite a lot.

The files people actually merge

Think about what PDFs people typically need to merge and why:

  • Multiple bank statements into a single document for a loan application
  • A signed contract and its annexures before sending to a counterparty
  • Payslips and employment letters for a rental application
  • Medical reports and referral letters before a specialist appointment
  • ID documents and supporting evidence for a legal or government submission

These are documents with real sensitivity. They contain full legal names, ID numbers, account numbers, salary figures, medical history, and home addresses. People use online tools for them because the alternative — buying a PDF editor — costs money, and because the process is fast and works on any device.

The argument that "these tools delete your files quickly" is true but incomplete. The question isn't just whether they delete your files — it's whether any of the following happened while your file was on their server: indexed by their search system, touched by an employee investigating a bug, included in a backup that outlasts the deletion window, or caught in a security incident. These things happen at reputable companies. They happen rarely. But they happen.

Why browser-based PDF merging is now genuinely possible

PDF is an open standard. The specification is publicly available, and open source libraries that can read and write PDFs have existed for years. What changed recently is that these libraries can now run inside a browser using JavaScript and WebAssembly.

The library Neatkit uses is pdf-lib — a pure JavaScript PDF toolkit that runs entirely client-side. It can merge PDFs, split them, rotate pages, and reorder them. It handles the PDF specification correctly, including embedded fonts, images, form fields, and digital signatures. There are no server calls in the merge process at all — the library reads your files from your device's memory, processes them in the browser's JavaScript engine, and writes the output back to memory for you to download.

This means the process works even if you're offline. After the page loads, you can disconnect from the internet entirely and the merge still works. That's a reliable test of whether a tool is genuinely browser-based: does it function without a network connection?

What you lose — and what you don't

Browser-based PDF merging is not a compromise for most use cases. For standard PDF files — documents with text, images, standard formatting — the output quality is identical to server-side processing. Both approaches are reading and writing the same PDF specification.

There are edge cases where browser-based processing has limitations. Very large files (several hundred megabytes) will be slower on older devices because the processing happens on your device's CPU rather than a server. PDFs with complex interactive elements like JavaScript-based forms may not transfer perfectly. Encrypted PDFs that require a password to open need to be decrypted first.

For the common use case — combining 2–10 standard PDF documents into one — there is no meaningful difference in output quality, and the privacy advantage is absolute.

How to verify a tool is actually browser-based

Tools can claim to be private without being browser-based. The word "secure" on a website is marketing, not a technical specification. Here's how to actually verify:

  • The offline test: Load the page fully, then disconnect from the internet. Drag in your PDFs. If the merge works, the processing is local. If you get an error or spinning wheel, it's server-side.
  • The DevTools test: Open your browser's developer tools (F12), go to the Network tab, and process your files. Look for any POST requests or requests that contain file data. A genuine browser-based tool will show no upload traffic.
  • The source code check: Some tools are open source. If the code is public, you can verify exactly what it does. Closed-source tools require trust.

Neatkit's PDF Merge tool passes all three. The source code for the pdf-lib library it uses is on GitHub. The offline test works. The Network tab shows no file uploads.

The account wall problem

Many PDF tools push users to create accounts after a few free merges. The reason is usually stated as "to save your files" or "for your security" — but the real reason is that server-side processing costs money, and free unlimited use isn't sustainable under that model.

Browser-based tools don't have this problem. There are no server costs associated with your merge operation, so there's no economic pressure to limit how many merges you can do. Neatkit has no usage limits, no account requirement, and no paid tier for PDF merging.

A practical workflow for sensitive documents

If you're regularly merging sensitive PDFs — for a legal practice, an accounting firm, an HR department, or personal financial documents — here's a workflow that keeps everything on your device:

  • Use a browser-based PDF merger for combining documents
  • Use a browser-based PDF compressor if the merged file is too large to email
  • Send via encrypted email or a secure file transfer rather than standard attachments for highly sensitive documents

The merger and compressor steps can both happen in the browser without any files leaving your device. What you do with the merged file after that is a separate question — but at least the processing step isn't creating unnecessary exposure.
